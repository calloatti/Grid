TIMBERBORN ENGINE & GRID ARCHITECTURE REPORT
System Context for Modding & Voxel Scanning Logic

1. The Bounding Box and BlockObjectSpec
Timberborn's grid does not support natively complex 3D shapes. Every single entity placed in the world—from a tiny shrub to a massive Monument—is defined by a strict 3D rectangular bounding box under the BlockObjectSpec -> Size {X, Y, Z} property.

To create complex visual shapes (like pyramids, arches, or L-shapes), the engine uses an array of Blocks that fills this bounding box layer by layer, from bottom to top.

2. Intra-Voxel Collision (Occupations)
Within a single 1x1x1 grid voxel, multiple objects can coexist as long as they do not claim the same Occupations. The game divides a voxel into sub-layers and flags:

All: Occupies the entire voxel. Nothing else can share this space.

Bottom / Floor: Claims the ground layer of the voxel.

Top: Claims the ceiling of the voxel (e.g., Overhang buildings use this, allowing a path to exist underneath them in the exact same Z-coordinate).

Path: A crucial collision flag. Even if a building is not solid, claiming "Path" prevents the player from routing dirt or wooden paths through that voxel (e.g., Thorns or factory floors).

Middle: Commonly used to block power shafts.

None: Negative Space. The engine treats this voxel as empty air, allowing the player to build paths, power shafts, or bridges through the building's bounding box.

3. Vertical Construction (Stackable)
In the game's C# API, checking obj.Solid is a shortcut for checking the blueprint's Stackable property. Stackable is not a boolean; it is an Enum that defines what can be built on top of the voxel:

None: Non-solid. (e.g., Shrines, Decorations, Floodgates, Roof edges).

BlockObject: Standard solid surface. You can build other standard buildings or platforms on top (e.g., Lodges, Warehouses, Levees).

UnfinishedGround: Acts identically to natural terrain. Objects that specifically require dirt/ground (like crops, trees, or Mud Baths) can be placed on top of these (e.g., TerrainBlock).

4. Entity Types & Components
The engine does not have a hardcoded "Building" flag. Entities are defined by the C# components attached to them in the blueprint:

Buildings: Have the Building component (costs resources, built by beavers).

Natural Resources: Have the NaturalResource component (Trees, Bushes, Crops).

Ruins: Have the Ruin component.

Map Editor Objects: Have no Building, NaturalResource, or Ruin component. They are just raw BlockObject entities (e.g., Blockage, UnstableCore, Slope).

5. Architectural Quirks & Edge Cases
The "Fermenter" Problem (Bounding Box Illusions): Many large buildings (Fermenters, Monuments, Earth Recultivators) are tall but have "Occupations": "None" in their upper layers to account for sloped roofs or spires. The 3D Mesh extends into these empty logical blocks.

The Verticality Trap: Dam blocks are Stackable: BlockObject (Solid). Floodgate blocks are entirely Stackable: None (Non-Solid), even though they look flat on top.

Partial Solidity: Buildings like Roof3x2 have BlockObject on their center ridge, but None on their slanted sides, meaning a single building can be partially solid.

Paths & Bridges: Structures with the Path component occupy the bottom of their assigned Z-voxel. A bridge at Z=1 has its wooden deck resting at World Y=1.0, not World Y=2.0.

6. The Golden Algorithm for Voxel Surface Scanning
Use this logic to build tools that need to find the visual "roof" or interaction layer of the grid (like Rulers, Grids, or Markers).

Because of the engine quirks above, scanning from bottom-to-top (column by column) requires specific handling:

Bypass Terrain Elements: Explicitly ignore any BlockObject that contains a NaturalResource or Ruin component. Treat them as air so tools project onto the ground beneath them.

Evaluate Solid vs. Non-Solid:

If an object is Solid, it becomes the definitive surface at that Z-level.

If an object is Not Solid, but has a Building component, it is tracked as a valid surface.

Active Bounding Box Tracking (The Memory System):

To solve the "Fermenter Problem", when the scanner hits a building at Z=0, it must remember that activeStructure.

If the scanner moves up to Z=1 and the engine reports "Empty Air" (Occupations: None), the scanner must check if Z=1 is still within the activeStructure.PositionedBlocks grid. If it is, the scanner treats that "Empty Air" as the building's roof to perfectly accommodate the 3D mesh.

Occlusion Checking:

Before rendering a marker on a non-solid roof or empty bounding-box voxel, check Z+1. If another solid object (like a bridge) was built in the "Negative Space" of the building, the bridge takes priority.
